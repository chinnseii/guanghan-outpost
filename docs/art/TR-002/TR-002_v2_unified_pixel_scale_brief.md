# TR-002 训练中控室 — V2 统一像素尺度重构指令

## 这一轮为什么存在

上一轮（"Full Reconstruction on the Shared Modular Atlas"）已经把地板/控制台/门框接入了游戏，
但美术验收没有通过。原话：

> 问题不只是"丑"，而是画面没有统一的像素尺度。现在像是三套素材硬拼：人物是清晰的小型像素角色；
> 地板是放大的高细节半写实金属贴图；控制台、墙体又是另一种更软、更高分辨率的绘制方式。最明显的
> 比例错误是：角色约高半块地板，但在俯视游戏里，角色应约高 1.2–1.5 个地砖。现在每块地砖太大、
> 细节太密，角色像缩小后贴在工业概念图上。

现状截图（**反面参考，新一轮不能重复这个效果**）：`docs/screenshots/training_hub_art/01_hub_panorama_no_ui.png`

看这张图就能确认问题：左右两台控制台是软阴影、发光屏幕的插画风格；地板是密集铆钉+划痕+磨损的
仿真金属贴图；而中间站着的角色是干净扁平的小型像素图——三者像是从三个不同项目里各拿一张图拼在
一起，不是同一个房间。

这一轮**不是继续微调现有素材**，而是按统一像素密度重新生成：地板、墙体、门、控制台必须和角色
用同一套"像素预算"绘制，做出来的效果应该是**清晰可读的 2D 像素地图**，不是工业概念场景图。

## 统一尺度契约（这是这一轮最重要的新规则）

| 项目 | 数值 | 说明 |
|---|---|---|
| 游戏逻辑地砖显示尺寸 | **64×64px** | 这是新的基准单位，下面所有尺寸都按它换算 |
| 角色显示高度 | **80–96px**（约 1.25–1.5 个地砖高） | 参考角色素材：`assets/characters/player_preview/female/light/black/walk_cycle_long.png` 或任意已接入的 walk_cycle 素材，此角色的像素密度/线条粗细是本轮所有环境素材必须匹配的基准 |
| 中央控制台显示宽度 | **96–128px**（约 1.5–2 个地砖宽） | 不能比这更宽，避免视觉上压过角色 |
| 每个表面的明暗层级 | **2–3 级**（暗部/中间色/高光，不做渐变） | 地板、墙体、门、控制台外壳统一执行 |
| 渲染风格 | 扁平像素填色 + 清晰轮廓线 | 不做柔焦、照片级高光、写实材质贴图、法线贴图式凹凸 |

**判断标准（换算方法）**：假设把地砖当成从一张 16×16 或 32×32 的原生小尺寸像素图放大来的（哪怕
实际出图分辨率更高），细节密度也不能超过这个"小图放大"的观感——不能出现只有在原图很大尺寸下才
看得清的细小划痕、铆钉纹理或渐变阴影。

**出图分辨率要求（技术约束，避免接入时二次踩坑）**：所有单件素材建议按 **64px 的整数倍**出图
（比如每块地砖原生 256×256 或 512×512），不要出一个"看着够用"但不是 64 整倍数的尺寸。原因：

1. 整数倍尺寸方便工程侧直接按比例缩小接入，不用再单独计算裁切偏移。
2. 这个项目在角色素材上踩过真实的坑——如果原生分辨率和最终显示尺寸的缩放比例太极端（超过
   ~9 倍），Godot 运行时直接绘制会出现发黑、锯齿的渲染问题（无 mipmap 的 GPU 缩小采样导致）。
   之前 512px 原生图缩到 56px 显示（约 9 倍）就出过这个问题，后来统一做法是：先离线把图缩小到
   安全比例（4~6 倍以内）再实际接入游戏。这次地砖/墙体/门/控制台如果按 256×256 或以内出图、
   最终显示在 64×64（约 4 倍），可以直接安全使用，不需要工程侧再做一次离线缩放。如果出图分辨率
   明显更高（比如 1024×1024 出到 64×64 显示，约 16 倍），工程侧会先离线缩小一版再接入，不会
   直接用原图。

## 分件规格

### 1. 地板 Tile

- 干净的扁平色块拼接，2–3 级明暗即可（比如：基础色 + 稍暗的接缝线 + 少量高光边），**不要**大面积
  铆钉、划痕、锈蚀、检修盖板这类高频细节——那些是上一轮被拒绝的"写实金属"风格。
- 保留少量变体是可以的（比如 1 种主板材 + 1 种带格栅的变体），但每种变体的细节密度必须和主板材
  一致，不能突然出现一块特别"精细"的贴花。
- 出图建议：单个 tile 按 64×64 或 32×32 原生分辨率设计（放大到 64×64 显示时刚好是 1:1 或 2:1，
  不会因为拉伸产生模糊插值边缘）。
- 存放路径建议：`assets/art/training_hub_v2/tiles/`

### 2. 墙体 + 墙角

- 和地板同一套明暗层级（2–3 级），深蓝灰基调，允许 1–2 条细的科技蓝装饰线，**不要**软光晕、
  不要大面积渐变阴影。
- 直墙 + 墙角各一套，可平铺/可旋转复用（沿用上一轮 `docs/art/TR-002/asset_prompts.md` §6.1 的
  尺寸建议即可，只是重新按新的明暗规则出图）。
- 存放路径建议：`assets/art/training_hub_v2/walls/`

### 3. 门（明确拆成 4 层，不再是一整块门框贴图）

上一轮的门是"门框贴图整体一张图，直接嵌进墙体"，这次要求拆成 4 个独立部件，方便以后单独换状态
（比如解锁/上锁变色）而不用重出整张图：

| 层 | 内容 | 备注 |
|---|---|---|
| 门框（frame） | 嵌入墙体的静态边框 | 深蓝灰，和墙体同色系 |
| 门体（body） | 门扇本身 | 可以只做"关闭"一个状态，锁定/解锁的变化交给代码层的颜色叠加，不需要美术做两套 |
| 门灯（light strip） | 门缝一条状态灯带 | 独立小图层，方便代码根据锁定状态换色（白/蓝=可用，红=锁定，已有的做法） |
| 方向铭牌（placard） | 图标牌（闪电/叶子/头盔/风扇） | **这 4 个上一轮已经生成过且本身风格是扁平图标，不在这次被拒绝之列——先直接复用 `assets/art/training_hub/props/door_sign_*.png`，除非贴到新门框上明显不协调，才需要重做** |

- 横向门框（配电房/训练温室，上下墙用）+ 纵向门框（宇航服整备室/空气系统控制室，左右墙用）各出
  一套，门框+门体可以合并成一张图（因为不需要开门动画），门灯单独一层。
- 存放路径建议：`assets/art/training_hub_v2/doors/`

### 4. 中央控制台

- 显示宽度控制在 96–128px（约 1.5–2 个地砖），不能像上一轮那样接近甚至超过 2 个地砖还要往上。
- 外壳用和墙体/地板一致的深蓝灰扁平色块，2–3 级明暗；屏幕部分可以保留科技蓝，但**不要柔光晕染
  的"发光屏幕"效果**——用一块纯色/双色块表示屏幕即可，可以加 1–2 个像素级的图表/文字占位符号，
  不需要精细插画。
- 存放路径建议：`assets/art/training_hub_v2/props/hub_console.png`

### 5. 顶部灯具

- 简单的像素灯具外壳 + 一条纯色/双色发光条，不做柔光晕染或镜头光晕效果。
- 存放路径建议：`assets/art/training_hub_v2/lighting/`

## 禁止事项（Do Not Drift）

- 不允许任何一件资源使用"写实/半写实金属材质"风格（渐变阴影、噪点划痕、法线贴图式凹凸）。
- 不允许控制台/墙体用比地板更"软"或更高分辨率的绘制方式——四类资源（地板/墙体/门/控制台）必须
  是同一支笔画出来的观感。
- 不允许中央控制台宽度超过 2 个地砖。
- 不允许把门做成一整块贴图糊住整个墙体开口——必须保持 frame/body/light 三层可拆分（铭牌已有，
  见上表）。
- 不允许引入霓虹、紫粉色、大面积暖光——延续 TR-002 原有 7 色板（见 `docs/art/TR-002/README.md`
  "Color Palette" 一节），只是这次要求所有资源在同一明暗层级下使用这套色板。

## 验收方法（生成后，工程侧会这样验证）

1. 把新地板 tile 按 64×64 铺出一个网格，叠加已接入的角色素材（原生渲染高度约 56px，会按新契约
   调整到 80–96px），确认角色高度落在 1.25–1.5 个地砖之间，不再出现"角色比地砖矮一半"的情况。
2. 把新控制台贴到网格上，量出它占几个地砖宽，确认在 1.5–2 个之间。
3. 截图对比新旧两版（新版应该明显更"扁平/干净"，旧版截图见上面的反面参考），确认地板/墙体/
   控制台/门看起来像同一支笔画的，不再是"三种拼贴"的观感。
4. 门的 frame/body/light 三层分别导入，确认锁定/解锁两种状态只需要改灯带颜色，不需要换整张门图。

## 关于 TR-001（训练模块一）的提醒

TR-001（宇航服整备室）目前也只有一张完整氛围参考图，从未被拆成独立小文件（见
`docs/art/TR-002/asset_prompts.md` §6 的说明）。也就是说 TR-001 很可能存在和这次 TR-002 完全
一样的"像素密度不统一"问题，只是这次审查的是 TR-002。等这一轮的统一像素尺度契约验证可行之后，
建议 TR-001 也按同一套契约重做，让训练区的所有房间在同一套视觉语言下——但这不是本轮要做的事，
先在这里记一笔，等你决定要不要一起排期。

## 交付后

按老规矩，把生成好的文件路径发回来，工程侧负责：
1. 按分层方式接入 `training_base_map.gd`（`_build_floor()`/`_build_structure()`/`_build_props()`
   等，具体函数名以当前代码为准）；
2. 跑 `--headless --editor --quit` 重新导入；
3. 隔离沙盒截图验证比例（角色/地砖/控制台的相对尺寸）、四个方向门的解锁/锁定状态、交互高亮不受
   影响；
4. 存档 SHA-256 前后比对，确认没有动到真实存档；
5. 截图发给你验收，不会自己判定 VISUAL_PASS。
